Wallet Signing Paths

Reference notes — how dApp signature requests and EOA transactions differ at the RPC level, and where Blockaid fits in each.

TL;DR. Wallets expose two fundamentally different signing paths. Drainers exploit Path A because it produces no on-chain activity for the victim and no gas popup. Direct transfers use Path B, which does go on-chain. Blockaid inserts a scan step before the wallet renders its confirmation popup on both paths — decoding the opaque payload into plain-English economic effect.

Path A — dApp Signature Flow (permit / drainer pattern)

The victim's wallet only sees two RPC calls. Neither produces any on-chain activity on the victim's side.

RPC 1eth_requestAccounts

  • Triggered when the user clicks Connect Wallet.
  • Returns the victim's address to the dApp.
  • No gas, no transaction, no blockchain footprint.

RPC 2eth_signTypedData_v4

  • Triggered by the site right after it has done recon on the victim's holdings.
  • Presents an EIP-712 typed-data payload (e.g. an ERC-2612 permit) for the user to sign.
  • Returns an ECDSA signature (v, r, s).
  • Still no gas, no transaction, no blockchain footprint on the victim's side.
Why this is the dangerous path
The attacker then uses that signature later, from their own wallet, paying their own gas, to execute permit() and transferFrom() on-chain. The victim never sees an eth_sendTransaction. No gas popup, no "Confirm transaction" screen, no pending tx in the activity tab — just a "Signature request" that users associate with benign wallet logins.

Path B — EOA Transaction Flow (direct transfer)

When Alice sends 500 USDT to Bob via MetaMask's native Send UI — no dApp involved — the wallet builds and broadcasts the transaction itself.

STEP 1MetaMask constructs the raw transaction object locally

{
  from:     "0xALICE...",
  to:       "0xdAC17F958D2ee523a2206206994597C13D831ec7",  // USDT contract
  value:    "0x0",                                          // no ETH being sent
  data:     "0xa9059cbb000000000000000000000000BOB00000...00000000000000000000001DCD6500",
  //         ^^^^^^^^                                                        ^^^^^^^^
  //         transfer(address,uint256) selector            amount: 500 USDT = 500_000_000 (6 decimals)
  nonce:    "0x2A",        // Alice's 42nd tx
  gasLimit: "0xEA60",      // ~60,000
  maxFeePerGas:         "0x59682F00",
  maxPriorityFeePerGas: "0x77359400",
  chainId:  "0x1"          // Ethereum mainnet
}

STEP 2MetaMask signs and RLP-encodes the transaction

raw_signed_tx = rlp_encode(
  nonce, gasPrice, gasLimit, to, value, data,
  v, r, s
)
// = 0xf86d2a843b9aca0082ea6094dac17f958d2ee523a2206206994597c13d831ec780b844a9059cbb...

STEP 3MetaMask broadcasts via eth_sendRawTransaction

The signed transaction hits the mempool, gets picked up by a validator, executed on-chain, and finalized. Alice sees -500 USDT, Bob sees +500 USDT. Total time ~12–15 seconds.

Where transaction simulation and validation fit
Before MetaMask shows Alice the confirmation popup (between Step 1 and the user clicking Confirm), this is the window where Blockaid's Transaction Scan API can simulate the transaction — trace its state changes, decode the calldata, flag unusual approvals or drains — and return a human-readable verdict the wallet renders in place of the opaque hex data field.

Key Contrast

Path A — dApp Signature Path B — EOA Transaction
RPC method eth_signTypedData_v4 eth_sendTransaction / eth_sendRawTransaction
Gas paid by victim? No Yes
On-chain activity from victim? No Yes
Wallet UI label "Signature request" "Confirm transaction"
Where Blockaid inserts Between the site's eth_signTypedData_v4 call and the wallet popup — scans the typed data, decodes the economic effect Between the eth_sendTransaction call and the wallet popup — simulates the transaction, decodes calldata
What Blockaid decodes into plain English "This permit will drain your entire USDT balance to an unknown address" "You will send 500 USDT to 0xBOB..."
The unifying insight
Both paths route through the same choke point for Blockaid: the wallet queries the Scan API before showing the user a popup, and renders the decoded verdict in place of (or alongside) the raw payload. The value is readability — Blockaid turns opaque typed data and hex calldata into "here's what will actually happen to your money," in both directions: flagging drains on malicious transactions, and giving confidence-inspiring clarity on legitimate ones.
