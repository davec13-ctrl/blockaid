Part 1 · Plain English

The five ideas everything is built on

Get these and the rest of the platform reads easily. A monitor watches your stuff; when its condition is met it opens an incident; the incident's severity sets how loud it is; a response is the automatic action that follows.

01

Inventory

The list of things you've asked Blockaid to watch — contracts, wallets, tokens, domains. Monitors point at these "assets."

02

Monitor

A rule: "watch X, and if Y happens, tell me." Two families — System (prebuilt, one instance each) and Custom (you build them).

03

Incident

What a monitor creates when it fires. This is the alert you triage.

04

Severity

How urgent: Info · Low · Medium · High · Critical. You choose it per rule.

05

Response

The automatic follow-up when a rule fires — Slack, email, webhook, or even an onchain transaction.

Part 1 · Plain English

The powerful rules, in everyday terms

These are the workhorses Tommaso walks through — the building blocks you'll reach for most. Orange badge = the four most flexible / powerful ones.

Core

Event Emitted

Like watching for a specific receipt to print.

Contracts announce what they do by firing events. This rule catches a named event the moment it's broadcast — the confirmation that something happened.

Watch for: an ownership-change event · a large withdrawal · a pause. Refine by event parameters.

Core

Function Call Detected

Like catching someone pressing a button — before the result shows.

Fires when a specific function is called on a contract. Catches the request/intent, where Event Emitted catches the confirmation.

Watch for: upgradeTo() called by anyone who isn't your admin wallet. Filter by function name, params, or caller.

Most powerful

Formula Rule

A smart calculator running over live data, 24/7.

You define metrics (a balance, a supply, a price, an API value, an event field) that get pulled continuously, then write a formula — a math/trend condition. When it's true, it fires. This is what does "value changed over time," which block-based tools can't.

Watch for: reserve ratio < 1.2 · TVL dropped 25% in 1h · supply up 50% in 24h.

Core

Proxy Upgrade Detected

An alarm when someone swaps the engine while the car keeps the same plates.

Upgradeable contracts can point to new code behind the scenes. This fires when a contract's implementation is changed by an address that isn't on your allowlist — a classic takeover signal.

Watch for: any proxy in inventory upgraded by a non-approved address.

Contract Owner Change

Alert when the keys to the building change hands.

Fires when a contract's ownership changes in a way that breaks your rules (e.g. the address doing it isn't allowlisted).

Watch for: ownership transferred by an unexpected modifier.

Composite Monitor

"X happened but Y didn't" — a broken-invariant detector.

Combines several monitors over a time window. Great for catching things that should always move together but didn't.

Watch for: owner changed (state) but setOwnership() was never called.

Delegate Call to Non-Approved Contract

Your contract borrowing code from a stranger.

Fires on a delegatecall to any implementation not on your approved list — indirect logic changes that are easy to miss.

Watch for: delegatecall to an address outside your approved implementations.

EIP-7702 Set-Code Upgrade

A wallet suddenly gaining a contract's powers.

Detects EIP-7702 upgrades touching a monitored wallet or implementation — a newer account-abstraction attack surface.

Watch for: a monitored wallet delegating to fresh code.

Asset Has Deployed a Contract

Watching who's building new things.

Fires when a monitored address deploys a new contract — useful for tracking a suspicious actor's footprint.

Watch for: a flagged actor deploying new tokens/contracts.

Rule of thumb. Reach for Event Emitted / Function Call Detected for single, specific onchain actions; reach for Formula Rule when the trigger is about a number, ratio, or trend over time; reach for Composite when the trigger is a relationship between two events.

The heavy-hitter system rules

You don't build these — Blockaid ships them, backed by ML and threat intel, and auto-scopes them to your inventory. One instance each; you tune, not create. These are the ones that catch an attack in progress.

System

Contract Was Exploited

The smoke alarm that goes off during the fire, not after.

Detects a successful exploit of one of your contracts in real time — the undesired activity that causes fund loss — so you're alerted while it's happening.

Fires when: an exploit is detected on a monitored contract.

System

Exploit Contract Deployed

Spotting the burglar building their tools before the break-in.

Blockaid's ML flags freshly deployed contracts that look like attack contracts aimed at your entities — often an early warning before the exploit executes.

Fires when: an exploit contract is deployed targeting your inventory.

System

Private Key Was Compromised

Someone else is now driving your car.

Detects clear signs that a wallet's private key is compromised and under unauthorized control — which usually means imminent, unauthorized fund movement.

Fires when: a monitored wallet shows key-compromise behavior.

System

Wallet Got Drained

The account gets emptied into a stranger's hands.

Detects fund loss from a monitored wallet to a malicious actor — the drain itself, so you can respond and trace immediately.

Fires when: a monitored wallet sends funds to a known malicious actor.

System

Funds Exposed to Malicious Actors

Handing a stranger a key to your safe — before anything's taken.

Detects token approvals/allowances granted to malicious actors. The exposure exists the moment the approval is signed, even if funds haven't moved yet — a chance to revoke first.

Fires when: a monitored entity approves funds to a malicious actor.

System

SUI Object Owner Change

The master keycard changes hands on Sui.

Detects unauthorized ownership changes on Sui capability objects (AdminCap, UpgradeCap) — the Sui equivalent of a privileged-role or ownership hijack.

Fires when: a monitored capability object moves to an unexpected owner.

System vs. custom, quickly. System rules answer "is a known attack pattern happening to me?" (Blockaid's detection). Custom rules — including Proxy Upgrade Detected and Contract Owner Change — answer "did this specific thing I care about happen?" (your logic + allowlists). Proxy Upgrade and Owner Change are custom, not system, even though they feel systemic.

Formula Rule — the deep end

Define metrics (data pulled continuously), optionally aggregate them, then write a formula. When the formula evaluates true, an incident fires. No backfill — a metric only has data from when you created it.

Metric types

TypeReads
Contract CallA numeric read-only function every block (balanceOf, totalSupply…)
Event (EVM & Sui)A numeric field from an event log (transfer value, swap amount…)
TransferTransfer amounts (USD) between addresses/tags
API CallA numeric field from an external HTTP endpoint (prices, custom API)
Read Object (Sui)A numeric field from a Sui object
Read Data Account (Solana)Token-2022 account state (mint supply, amount)
Instruction (Solana)Aggregated numeric param over Token-2022 instructions in a window
Computed MetricA new metric built from other metrics (e.g. TVL = A*B + C*D)

Aggregators (over a time window)

Sum · Count · Average · Minimum · Maximum · Standard Deviation

Formula syntax

GroupOperators
Math+ - * / ^
Compare> < >= <= == !=
Logicand · or
RangeA between 10 and 100
FunctionsABS(...) · TIMESTAMP_OF(A)
Trend (1 metric)increased_by · decreased_by · net_increased_by · net_decreased_by
Time units30s · 15m · 12h — values as %, x, or absolute

Worked examples

GoalFormula
Reserve ratio floorA / B > 0.9
TVL crashE decreased_by 25% in 1h
Balance surge(A > 10000) and (A increased_by 50% in 30m)
Pool imbalance(A / B > 1.2) or (B / A > 1.2)
Stale oracleTIMESTAMP_OF(A) - A > 3600
No backfill. Metrics only carry data from the moment they're set up — you can't query history before that. If a POV needs historical data, it's a custom engineering conversation, not a self-serve feature. Trend operators act on a single metric (no A increased_by B%).
