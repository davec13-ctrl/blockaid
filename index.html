⚡
Why pre-confirmation interception matters

Once a transaction hits the blockchain it is irreversible. Blockaid's value is entirely in Steps 2–5 — the window between the dApp requesting a transaction and the user confirming it. This is the only moment where a bad transaction can be stopped. Post-confirmation monitoring (watching events on-chain) can detect attacks but cannot reverse them.

Key Concepts in the Flow
📦
MempoolPre-confirmation waiting room

The pending transaction pool where signed transactions wait to be picked up by a validator/sequencer. Public — anyone can see pending txs. Source of MEV and front-running attacks.

🌍
World StateCurrent ledger snapshot

The complete current state of all accounts and contracts at the latest block — balances, nonces, contract storage (mappings). Blockaid reads this to simulate outcomes.

🔐
EIP-712 / Typed SigningOff-chain signatures

Structured off-chain message signing (used in Safe, Permit2, Seaport). Creates no on-chain transaction until the signature is submitted. Blockaid scans eth_signTypedData calls too.

⛽
Gas & Gas LimitTransaction cost control

Gas = computational work. Gas limit = max you'll pay. If a tx hits the limit it reverts (fails) but still costs gas. Unusually high gas limits can signal complex/malicious contract calls.

🔢
NonceReplay protection counter

An incrementing counter per account. Each tx must use the next nonce in sequence. Prevents the same signed transaction being broadcast twice. Safe also has its own internal nonce.

📋
ABI EncodingHow parameters are packed

Application Binary Interface — the encoding standard for calldata. Each parameter is padded to 32 bytes. Blockaid decodes ABI to extract exact addresses, amounts, and token types from raw hex.

JSON-RPC Methods — What Blockaid Monitors
📡
JSON-RPC is the interface between every dApp/wallet and the blockchain

Every action — from reading a balance to sending a transaction — is a JSON-RPC call. Blockaid's SDK intercepts these calls at the wallet layer before they reach the network, giving it full visibility into what a dApp is trying to do.

eth_sendTransaction
Sends a transaction to the network. The most critical call — creates on-chain state changes.
{ "to": "0x833589...", "value": "0x0", "data": "0xa9059cbb0000...7aa28...000186a0" }
📌 Blockaid decodes: function selector → token transfer · destination address · amount
HIGH PRIORITY — any token transfer, approve, or contract interaction flows through here
eth_signTypedData_v4 (EIP-712)
Signs structured off-chain data. Used by Permit2, Safe multisig, Seaport (OpenSea), and other protocols. Does NOT create a transaction — but the signature can be used later to authorize on-chain actions.
{ domain: { chainId, verifyingContract }, message: { spender, amount, deadline, nonce } }
📌 Blockaid decodes: who is being authorized · how much · expiry time · is the verifyingContract known?
HIGH PRIORITY — phishing sites abuse this to steal unlimited token approvals with no gas cost to victim
eth_sign / personal_sign
Signs arbitrary raw bytes or a message. Older method — less structured than EIP-712. Often used in authentication but also abused in signature phishing attacks.
📌 Blockaid flags: is this a known auth pattern or a suspicious raw hash that could authorize a transfer?
HIGH PRIORITY — "blind signing" — user cannot read what they're signing
eth_call
Read-only call to a contract. Does not create a transaction, costs no gas. Used to read state: balances, allowances, prices. Blockaid uses this internally to simulate transactions against live state.
eth_call({ to: USDC, data: "0x70a08231" + address }) → returns balance
📌 Blockaid uses this: reads world state before simulation · checks allowances · verifies contract existence
READ ONLY — no state change possible, but used by Blockaid as its eyes into the chain
eth_sendRawTransaction
Broadcasts a pre-signed transaction to the network. The wallet signs the tx locally, then uses this to push it to a node. This is the point of no return — once submitted, it enters the mempool.
📌 Post this point Blockaid can only monitor (not prevent). Pre-signing interception is critical.
BROADCAST — irreversible once it hits the mempool
wallet_switchEthereumChain / wallet_addEthereumChain
dApp requests to switch the wallet to a different chain or add a new custom chain. A common attack vector — malicious dApps add fake chains that mimic mainnet to steal funds.
📌 Blockaid checks: is this a known ChainID? Does the RPC URL match known providers? Is the chain verified?
HIGH PRIORITY — chain spoofing attack vector
eth_getTransactionReceipt
Fetches the receipt of a confirmed transaction — includes status (success/fail), gas used, and all event logs. Used post-confirmation to verify outcomes and extract emitted events.
📌 Contains the Logs array — all events emitted during execution. Source of truth for what actually happened.
READ ONLY — post-confirmation monitoring tool
eth_getLogs
Queries all events emitted by contracts matching a filter (address, topic, block range). How block explorers and indexers build their databases. Blockaid uses this for on-chain monitoring.
📌 Filter by topic[0] = Transfer event hash to find all token movements. Filter by topic[2] = address to find all approvals granted to a spender.
MONITORING — used for post-confirmation threat intelligence
Function Selectors — First 4 Bytes of Calldata
💡
How to read a function selector

Every transaction's data field starts with a 4-byte selector = keccak256("functionName(types)")[0:4]. This tells the contract which function to run. Blockaid identifies the intent of any transaction in milliseconds by matching these selectors against a known database.

0x23b872dd transferFrom(address,address,uint256) ERC-20 Token Spender pulls tokens from owner's wallet (using existing allowance). How DEXs actually move your tokens. HIGH RISK

⚠️
The most dangerous selector combinations for Blockaid to flag

approve + unknown contract — granting unlimited allowance to an unverified or newly deployed contract is the #1 phishing vector. setApprovalForAll + marketplace — if the marketplace isn't OpenSea/Blur/known, this empties your entire NFT collection. eth_signTypedData with no known verifyingContract — blind Permit2-style signature that could grant unlimited spending rights.

Blockaid Risk Signals — What to Monitor and Why

🔴 High Risk Signals

  • Unlimited approval to unverified contract
  • setApprovalForAll to unknown address
  • eth_signTypedData with unknown verifyingContract
  • Newly deployed contract (<24h old)
  • Simulation shows 100% asset drain
  • Known drainer contract address
  • Fake chain RPC (wallet_addEthereumChain)
  • transferFrom pulling all tokens
  • Permit signature with max uint256 value

🟡 Medium Risk Signals

  • Approval to known protocol but unusual amount
  • Swap with very high slippage (>10%)
  • Contract not verified on block explorer
  • Interaction with contract on different chain than expected
  • Unusual gas limit (too high)
  • Token not in known token list
  • Front-running / sandwich MEV exposure
  • Simulate shows unexpected token leaving wallet

🟢 Low Risk / Benign

  • Transfer to known address with expected amount
  • Approve to Permit2 (known contract)
  • Swap on Uniswap/verified DEX within expected range
  • Read-only calls (eth_call, balanceOf)
  • WETH wrap/unwrap
  • Known NFT marketplace interaction
  • Safe multisig execTransaction with known signers
The Simulation Pipeline
1️⃣
Decode CalldataWhat function? What params?

Read function selector (first 4 bytes). ABI-decode parameters. Identify: token address, spender, amount, deadline. For complex txs (Uniswap execute) — recursively decode nested commands.

2️⃣
Read World StateCurrent ledger via eth_call

Query current balances and allowances. Check contract bytecode — is it verified? Is it new? Compare to known contract database (Uniswap, USDC, Safe addresses). Flag if unknown.

3️⃣
Simulate Executioneth_call with state override

Run the transaction against current world state without broadcasting. Capture exact state changes: which balances increase/decrease, which approvals are set, which events are emitted.

4️⃣
Score & ReturnRisk label + decoded outcome

Combine: decoded intent + simulated outcome + contract reputation + known patterns → BENIGN / WARNING / MALICIOUS score. Return human-readable description: "This will transfer 100 USDC to 0x..."

Key Blockaid Concepts for SE Conversations
🧪
Simulation vs. Validation

Simulation = running the tx and seeing what state changes result. Validation = checking if the tx follows rules (correct nonce, sufficient balance). Blockaid does both — simulation catches unknown attack patterns validation would miss.

📦
Asset Exposure

Total value at risk from all outstanding approvals. A user might have forgotten they approved Uniswap V2 for unlimited USDT 2 years ago — that approval is still live on-chain. Blockaid can surface and quantify this.

🔗
Cross-chain Monitoring

Each EVM chain has separate state. An approval on Ethereum doesn't exist on Base. Blockaid monitors each chain independently, using chain-specific RPC nodes and contract databases per ChainID.

🎭
Contract Spoofing

A malicious contract can have the same name/interface as USDC but be entirely different code. Blockaid checks the contract address — not the name. Only the address is the ground truth.

⏱
Real-time vs. Batch

Pre-confirmation scanning must return in <500ms (user is waiting). Post-confirmation monitoring (watching mempool and blocks for threats) can be async. Blockaid operates at both layers.

🔄
Revoke vs. Reduce

Revoking = setting allowance to 0. Reducing = lowering to exact needed amount. Both require on-chain transactions (gas). Blockaid can recommend revokes as part of a risk remediation flow.

SE Terminology Reference — Day-to-Day Client & Prospect Conversations
💼
How to use this glossary

These are the terms clients and prospects will use when discussing security posture, integrations, and incident response. Understanding these at a technical level lets you speak credibly about what Blockaid does and where it fits in their stack.

EOA — Externally Owned Account
Wallet
A standard wallet address controlled by a private key (MetaMask, Ledger). No code — can only initiate transactions. Contrast with smart contract accounts (Safe, ERC-4337).
Client says: "We need to protect our users' EOA wallets from phishing"
Smart Contract Account (SCA)
Wallet
A wallet that is itself a smart contract — like Safe multisig or ERC-4337 wallets. Has programmable rules: spending limits, multi-sig requirements, session keys, recovery mechanisms.
Prospect says: "We use Safe for our treasury" — they need execTransaction monitoring
Approval / Allowance
ERC-20
Permission granted by a token holder allowing another address (spender) to move their tokens. Lives in the token contract's allowance mapping. Persists until revoked.
Client concern: "Users are approving unlimited amounts to our contracts — is that safe?"
Permit2
Uniswap
Uniswap's universal approval contract. Users approve tokens once to Permit2 (0x0000...2BA3), then sign off-chain EIP-712 messages per transaction. Reduces on-chain approve calls.
You spotted this in the swap exercise — the approve → sign → execute flow
MEV — Maximal Extractable Value
Protocol
Profit extracted by reordering, inserting, or censoring transactions. Includes front-running (inserting before your tx), sandwiching (wrapping your tx), and back-running. Sequencers capture MEV on L2.
Prospect: "Can Blockaid detect sandwich attacks before they happen?"
Front-running
Attack
Observing a pending transaction in the mempool and submitting your own transaction with higher gas to execute first, profiting from the known outcome of the pending tx.
Common in DEX swaps — bot sees your large swap, buys first, sells after your tx moves the price
Drainer Contract
Attack
A malicious smart contract disguised as a legitimate dApp. When you "approve" it or sign a Permit message for it, it immediately drains all tokens from your wallet.
Phishing sites often prompt users to "claim NFT" which is actually approving a drainer
Signature Phishing
Attack
Tricking a user into signing a message (eth_signTypedData or eth_sign) that authorizes a transfer or approval — without the user realizing. No on-chain tx needed from the attacker initially.
Prospect: "Our users are getting drained after signing something on a Discord link"
Gas Griefing
Attack
Causing a transaction to consume more gas than expected, potentially causing it to fail (out of gas) or costing the user significantly more than anticipated.
Malicious contracts can loop or waste gas to drain ETH from the calling wallet
Honeypot Token
Attack
A token where buy transactions succeed but sell transactions revert or are blocked. Contract code prevents selling. Users get trapped holding worthless tokens.
Simulation reveals: the transfer() function has a blacklist that blocks all sells
Rug Pull
Attack
Project founders drain liquidity or mint unlimited tokens suddenly, causing the token price to collapse. Contract has backdoor mint functions or ownership of liquidity pool.
On-chain monitoring: detect sudden large liquidity removals or unexpected large mints
ABI — Application Binary Interface
Protocol
The encoding standard for interacting with smart contracts — defines how function calls and responses are serialized into bytes. Required to decode calldata and event logs.
Blockaid uses ABIs to decode what 0xa9059cbb0000... actually means in human terms
Events / Logs
Protocol
Permanent on-chain records emitted by contracts during execution. Have Topics (indexed, searchable) and Data (non-indexed). Source for all balance history, approval history, swap history.
Transfer + Approval events are the primary data source for Blockaid's threat intelligence
Mempool
Protocol
The pending transaction pool — txs that have been broadcast but not yet included in a block. Public and visible to all nodes. Source of MEV and front-running.
Prospect: "Can you monitor the mempool for transactions targeting our contracts?"
Nonce
Protocol
Transaction sequence counter per account. Prevents replay attacks. Must increment by 1 each tx. A stuck nonce (e.g., pending tx with low gas) blocks all subsequent transactions.
Client issue: "User's transactions are stuck" — likely a pending tx with too-low gas at the same nonce
CCTP — Cross-Chain Transfer Protocol
Bridge
Circle's native USDC bridge — burns USDC on source chain, mints fresh native USDC on destination. Not a wrapped token. Supported across Ethereum, Base, Arbitrum, Polygon, etc.
You used this in the exercise — Superbridge wraps CCTP for the user
Sequencer
L2
The centralized server on L2 chains that orders and batches transactions before posting them to L1. Controls MEV on L2. Run by the chain operator (Coinbase for Base, Offchain Labs for Arbitrum).
Prospect: "We're deploying on Base — who controls transaction ordering?"
ERC-4337 — Account Abstraction
Wallet
A standard enabling smart contract wallets without protocol changes. Users submit UserOperations instead of transactions. Enables: gasless txs (paymasters), batched calls, social recovery, session keys.
MetaMask Smart Account uses ERC-4337 — caused the $1.14 gas issue in your exercise
Slippage
DEX
The difference between expected and actual swap price due to pool price movement during tx execution. High slippage tolerance makes you vulnerable to sandwich attacks. Low slippage can cause tx to revert.
Uniswap shows "Price Impact" and lets you set max slippage tolerance
Token Decimals
ERC-20
ERC-20 tokens store balances as integers. Decimals define the divisor. USDC/USDT = 6 decimals (1 USDC = 1,000,000). ETH/most tokens = 18 decimals (1 ETH = 10¹⁸ wei). Always check decimals when reading raw amounts.
Raw value 978486 in USDC = 0.978 USDC (÷ 10⁶). In ETH = 0.000000000000978486 ETH (÷ 10¹⁸)
