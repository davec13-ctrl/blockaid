Executive Overview

Wallet Drainers: The Modern Threat to Signing Surfaces

As protocol-layer exploits have become harder, attackers have shifted to the softest remaining surface: end users. Wallet drainers trick users into signing transactions or messages they don't understand — turning a single signature into a full wallet drain. This is a short overview of how the threat works and how Blockaid detects it.

How Drainers Reach Users

Drainers are a product category with real engineering investment. Operators don't write malicious code — they buy it, then focus on distribution.

01
Kit Purchase
Operator buys a drainer SDK — JavaScript components that generate malicious transactions.
02
Brand Impersonation
Clone of a known dApp, often with convincing copy and verified-looking contracts.
03
Distribution
Paid ads on search & social, phishing email, DMs, and compromised community channels.
04
Signature Capture
User lands, connects wallet, and is prompted to sign a drainer-crafted tx or message.
05
Drain
Funds are swept — immediately on transfer drains, later on approvals and permits.

The SDK economy changes how we defend

Because the same kits are reused across hundreds of sites, static fingerprints of their JavaScript and network behavior let us flag new malicious dApps before a single user signs. Drainer kits also advertise "Blockaid bypass" claims in their sales channels — a signal we actively monitor.

The Six Drainer Categories

Severity: High — immediate drain Medium — delayed drain Context-dependent
01 · Native Transfer

Wrong-Address Transfer

User signs what looks like a normal ETH/SOL/BNB transfer, but the recipient is an attacker-controlled EOA. Calldata is empty, so there is nothing to parse — detection depends entirely on counterparty reputation.

Key signals: recipient age on-chain, distinct-sender count, originating dApp reputation, value relative to wallet balance.
High severity Native asset EOA recipient
02 · Token Transfer

Transfer Farming

ERC-20 analog of the above. The dApp prompts an outright transfer of the user's token balance — often the full holdings — to an attacker address.

Key signals: recipient reputation, percent of user's token balance transferred, unusual batching across tokens.
High severity ERC-20 Full-balance drain
03 · Allowance Abuse

Approval Farming

User grants a malicious spender an approval — often unlimited. Nothing leaves the wallet in the moment, which is exactly why it works. The drain happens at the attacker's convenience.

Key signals: spender is an EOA (no code to audit), spender reputation, unlimited allowance, no matching legit protocol.
High severity Approval exposure Delayed drain
04 · Off-Chain Signature

Permit / EIP-712 Farming

Stealthier variant of approval farming. Instead of an on-chain approval, the user signs an EIP-712 message (e.g., permit) off-chain. Many scanners ignore signed messages entirely.

Key signals: Blockaid simulates the on-chain transaction the signature would enable and evaluates it like any other approval.
High severity EIP-712 Off-chain
05 · Contract Disguise

Stealthy Malicious Contracts

A contract with verified, human-readable source is deployed to add a veneer of legitimacy. The trick hides in plain sight — e.g., a claim function that credits an arbitrary recipient parameter instead of msg.sender.

Key signals: simulated balance changes don't match user intent, arbitrary-sender patterns in calldata, novel contracts surfacing across known malicious dApps.
High severity Verified source Semantic trick
06 · UI Layer

Address Spoofing

Because wallets truncate long addresses, attackers generate vanity addresses whose visible portion (first/last characters) matches a familiar counterparty — or the burn address. A UI-layer attack that signature review alone cannot catch.

Key signals: prefix/suffix collision with prior counterparties, lookalike of zero-address, transfer-to-self patterns that aren't actually self.
Medium severity UX exploit Vanity address

Why Two Detection Layers

Transaction scanning and dApp scanning are not substitutes — they reinforce each other. Used together, they close blind spots that single-layer defenses leave open.

Transaction Scanning

Simulates the actual effects of a signature — asset diffs, approvals, exposures, and exit paths — rather than parsing calldata.

  • Catches raw transfers, approvals, permits, and semantic tricks like arbitrary-recipient contracts.
  • Covers both on-chain transactions and off-chain EIP-712 messages.
  • Reputation context on every address and token involved.

dApp Scanning

Evaluates the site itself — JavaScript fingerprints, network behavior, and brand-impersonation signals — before any transaction is signed.

  • Flags drainer-SDK reuse across thousands of sites.
  • Surfaces malicious dApps even when the transaction they emit looks superficially clean.
  • Provides context that lifts transaction-layer confidence in ambiguous cases.

The feedback loop

When a known-malicious dApp emits a transaction that looks benign, we investigate a potential transaction-rule bypass. When a reputable dApp emits a transaction that looks malicious, we investigate a potential compromise. Neither layer alone produces this signal — the combination is what keeps pace with weekly drainer iteration.

What This Means for a Wallet Team

Simulation is not optional

A calldata-only view cannot distinguish an empty transfer to a malicious EOA from a benign send, cannot surface off-chain permit risk, and cannot catch stealthy contract patterns. Simulation is the baseline for meaningful user warnings.

Reputation scales where rules don't

Drainer kits change weekly; reputation signals accumulated across millions of wallet interactions do not. The breadth of Blockaid's integrations is a direct input into detection quality.

Both layers, or neither

Shipping transaction scanning without dApp scanning (or vice versa) leaves coverage gaps that drainers actively exploit. The two should be deployed as a package.

Feedback matters

A reporting endpoint for false positives and false negatives is part of the product. User- and integrator-reported cases feed directly into our threat intel pipeline.
