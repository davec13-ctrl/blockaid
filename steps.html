1
🌐 dApp → 🦊 Wallet

dApp sends eth_sendTransaction to Wallet

The user clicks a button on the web frontend (e.g. "Swap", "Approve", "Mint"). The dApp constructs a transaction object and hands it to the wallet extension via a JSON-RPC call. Nothing has been signed yet — the wallet hasn't even shown the user a popup yet.
What happens in detail

When you interact with a dApp — say you click "Swap 100 USDC for ETH" on Uniswap — the dApp's JavaScript does not directly write to the blockchain. It can't — only the wallet holds your private key. Instead, the dApp calls window.ethereum.request(), which is an API injected into your browser by MetaMask (or any wallet extension).

This call passes a raw transaction object: the destination contract address (to), the ETH value being sent (value), the encoded function call in hex (data), and optionally gas hints. The dApp says "please send this transaction" — but the wallet gets to decide whether to show it, block it, or ask the user.

The data field is the most critical part. It contains the function selector (first 4 bytes = which function to call) followed by ABI-encoded arguments (padded 32-byte chunks). This is how the contract knows what to execute.

The JSON-RPC call
// dApp JavaScript — running in your browser tab window.ethereum.request({ method: "eth_sendTransaction", params: [{ from: "0xYourWalletAddress", to: "0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48", // USDC contract value: "0x0", // no ETH sent data: "0xa9059cbb" // transfer() selector + "000000000000000000000000RecipientAddress..." // param 1: to + "0000000000000000000000000000000000000000000000000000000005F5E100", // param 2: amount (100 USDC) gas: "0xEA60" // gas limit hint }] })
Key fields decoded
data field — function selector
0xa9059cbb
First 4 bytes of keccak256("transfer(address,uint256)"). This tells the contract which function to execute. Blockaid reads this immediately to know what kind of action is being requested.
to field
Contract address
The smart contract being called. Could be a token contract, DEX router, NFT, or a malicious drainer. Blockaid checks this address against its known contract database.
value field
ETH amount (in wei)
Native ETH being sent with the call. 0x0 for ERC-20 token operations. Non-zero if ETH is being transferred or used to pay for something (e.g. paying mint price in ETH).
from field
Your wallet address
The EOA (Externally Owned Account) that will sign the transaction. The wallet uses this to know which private key to use for signing at Step 7.
🔑
Why the dApp can't skip the wallet

The dApp can read from the blockchain directly using eth_call — for example, checking your token balance. But to write (change state), a transaction must be cryptographically signed by a private key. That private key lives only in the wallet — never in the dApp. So every state-changing transaction must pass through the wallet.

💼 SE Talking Point
  • The wallet is the trust boundary between the dApp and the blockchain. Blockaid's integration sits at this boundary — it evaluates every transaction before the user even sees the popup.
  • A malicious dApp can craft a data field that looks like a simple transfer but is actually calling setApprovalForAll(attacker, true). Only decoding the calldata reveals this — which Blockaid does at Step 2.

🦊 Wallet → 🛡 Blockaid

Wallet calls Blockaid — scan_transaction(rawTx)

Before showing the user any confirmation popup, the wallet SDK silently forwards the raw transaction to Blockaid's API. This is the entry point of the entire security pipeline. The user has no idea this is happening — it's seamless and happens in milliseconds.
What happens in detail

Wallets that integrate Blockaid (MetaMask, Rainbow, Phantom, Zerion, and others) include the Blockaid SDK. When the wallet receives an eth_sendTransaction request at Step 1, it doesn't immediately render the confirmation UI. Instead, the SDK intercepts the request and fires an API call to Blockaid's simulation engine with the full transaction payload.

The Blockaid API receives the raw transaction — to, from, data, value, chainId — and begins its analysis pipeline. At this stage Blockaid has the encoded calldata but needs to simulate execution to understand what will actually happen on-chain.

This is also where EIP-712 typed data signing requests are intercepted — for example Uniswap's Permit2 off-chain signatures. These never touch the mempool, so the only security check is at this step.

What Blockaid receives
// Blockaid API payload — sent by Wallet SDK { "chain": "ethereum", "account_address": "0xYourWalletAddress", "transaction": { "from": "0xYourWalletAddress", "to": "0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48", "value": "0x0", "data": "0xa9059cbb000...encoded args..." }, "metadata": { "domain": "app.uniswap.org" // dApp origin URL } }
🛡 Blockaid immediately decodes

The function selector 0xa9059cbb is matched to transfer(address,uint256). The arguments are ABI-decoded to extract the recipient address and amount. Blockaid then knows: "this is a token transfer, not an approval" — and proceeds to simulate it.

⚡
Why "before the popup" matters

If Blockaid only ran after the user saw the popup, users would be making decisions without risk information. By running during Step 2 (before Step 6's popup), Blockaid can annotate the popup with a risk label, block it entirely for MALICIOUS results, or let it through cleanly for BENIGN results — all before the user has to act.

💼 SE Talking Point
  • Blockaid's SDK is embedded in the wallet, not the dApp. This means even if the dApp is compromised (e.g. its CDN was hacked), Blockaid still scans every transaction before the user signs.
  • This is why wallet integrations are Blockaid's primary distribution channel — the wallet is the last chokepoint before irreversible on-chain actions.
Step 2 of 10
3
🛡 Blockaid → ⚡ RPC Node

Blockaid simulates the transaction via eth_call

Blockaid calls eth_call on a blockchain RPC node to read the current world state and simulate what the transaction would do if executed right now — without actually submitting it. This is a read-only operation: no gas, no signature, no on-chain effect.
What happens in detail

eth_call is the blockchain equivalent of a "dry run". It lets anyone execute a transaction against the current state of the chain without broadcasting it. Miners/validators don't see it, no gas is consumed, and the state doesn't change.

Blockaid uses this to ask: "if this transaction were included in the next block right now, what would happen?" The EVM runs the code virtually against the live world state and returns the execution result — including any state changes, token movements, and emitted events.

Blockaid may call eth_call multiple times in this step: once to run the full simulation, and additional calls to read specific storage slots — like balanceOf(user) before and after, or allowance(owner, spender) to understand existing approvals.

🌍
World State = the current blockchain ledger snapshot

Every EVM simulation runs against the "world state" — the complete record of all account balances, nonces, and contract storage at the latest block. For ERC-20 tokens, a user's balance is just a value in the token contract's mapping(address => uint256) balances storage. Blockaid reads this mapping via eth_call to know what the user actually holds right now.

💼 SE Talking Point
  • Blockaid's simulation is not just "checking a list" — it actually re-runs the EVM bytecode against the live chain. This means it catches novel attack patterns that have never been seen before, because the outcome (e.g. "user's entire balance drained") speaks for itself.
  • For complex DeFi transactions with multi-hop routing (e.g. USDC → WETH → FAI via Uniswap), the simulation traces every internal call and sub-call to capture the complete token flow.
Step 3 of 10
4
⚡ RPC Node → 🛡 Blockaid

Node returns world state + simulation result

The RPC node responds with the full simulation output: what the current balances are, what the transaction would do to those balances, what events would be emitted, and whether the transaction would succeed or revert. This is the raw data Blockaid's risk engine processes.
What happens in detail

The RPC node (Alchemy, Infura, or a direct node) executes the simulated transaction and returns a detailed trace. This includes the final return value, all storage slot changes (state diff), all events that would have been emitted, and any internal calls made by the contract to other contracts.

For a token transfer, the state diff is simple: sender's balance decreases, recipient's balance increases. For a complex DEX swap, there might be dozens of internal calls and multiple token movements. For a drainer, the simulation might show the user's entire wallet contents being swept to an unknown address.

This is where Blockaid's proprietary intelligence layer matters most — it doesn't just receive the raw result, it interprets it through trained models that recognize attack patterns.

Example simulation result — Uniswap swap
// Simulation result from RPC node (simplified) { "success": true, "gas_used": 185000, "state_diffs": [ { "token": "USDC", "address": "0xYours", "change": -100000000 }, // -100 USDC { "token": "ETH", "address": "0xYours", "change": +42000000000000000 }, // +0.042 ETH { "token": "USDC", "address": "0xPool", "change": +100000000 } // pool receives USDC ], "events": [ { "name": "Transfer", "from": "0xYours", "to": "0xPool", "amount": 100000000 }, { "name": "Transfer", "from": "0xWETH", "to": "0xYours", "amount": 42000000000000000 }, { "name": "Swap", "sender": "0xRouter", "recipient": "0xYours" } ] }
What Blockaid's risk engine analyses
Balance delta
Net gain/loss per asset
Does the user end up with more or less of each asset? A drainer would show the user losing tokens to an unknown address with nothing received in return.
Recipient analysis
Who receives value?
Is the recipient a known DEX router, or a fresh wallet created 2 hours ago? Newly created recipient addresses with large inflows are a strong drainer signal.
Approval check
What is being authorized?
If the tx is an approve(), Blockaid checks: who is the spender? Is it a known protocol? Is the amount type(uint256).max (unlimited)? Is this spender on a block list?
Revert detection
Will the tx fail?
If the simulation reverts, Blockaid can warn the user before they waste gas on a transaction that would fail. Common cause: insufficient balance or stale allowance.
⚠️
Attack example: what a drainer simulation looks like

State diff shows: user's USDC balance: -50,000 USDC. User's ETH balance: -2.5 ETH. Recipient (unknown address, created yesterday): +50,000 USDC, +2.5 ETH. Zero assets returned. This is the signature of a wallet drainer — and Blockaid returns MALICIOUS before the user even sees the popup.

💼 SE Talking Point
  • This is the step where Blockaid's simulation-based approach separates it from simple signature-based security tools. Instead of just checking "is this contract on a blocklist?", Blockaid asks "what will actually happen to the user's assets?"
  • The RPC node Blockaid queries is its own infrastructure — not the same node the wallet uses to broadcast transactions. This ensures Blockaid always reads the most current world state.
Step 4 of 10
5
🛡 Blockaid → 🦊 Wallet

Blockaid returns risk_score + decoded_tx

After running the simulation and risk analysis, Blockaid sends back a structured response to the wallet. This includes a risk verdict (BENIGN / WARNING / MALICIOUS), a human-readable explanation, and the decoded transaction summary (what will change, in plain language).
What happens in detail

The Blockaid API response is the culmination of Steps 2–4. It contains the risk classification, the reason for that classification, the decoded transaction in human-readable form, and the expected asset changes. The wallet SDK receives this and uses it to build the confirmation popup shown in Step 6.

The three verdict types are: BENIGN (safe, proceed normally), WARNING (unusual but not necessarily malicious — show the user with caution indicators), and MALICIOUS (confirmed attack — wallet blocks the action and shows a red warning screen).

Blockaid API response structure
// Blockaid response — sent back to Wallet SDK { "status": "BENIGN", // BENIGN | WARNING | MALICIOUS "result_type": "Transaction", "description": "Swap 100 USDC for ~0.042 ETH via Uniswap V3", "simulation_results": { "account_summary": { "assets_out": [{ "asset": "USDC", "amount": "100.00" }], "assets_in": [{ "asset": "ETH", "amount": "0.042" }], "total_usd_out": "100.00", "total_usd_in": "99.70" } }, "classification_data": { "type": "KNOWN_MALICIOUS_ADDRESS", // only if WARNING/MALICIOUS "reason": "Recipient is a known drainer contract" } }

🛡
Steps 2–5 is Blockaid's entire value window

This four-step loop — intercept → simulate → analyse → score — all happens in under 300ms before the user sees the popup. Once a transaction is signed and broadcast (Step 8), it is in the mempool and heading for the blockchain. There is no recall. Blockaid's protection is entirely front-loaded into this pre-confirmation window.

💼 SE Talking Point
  • The description field ("Swap 100 USDC for ~0.042 ETH via Uniswap V3") replaces the raw hex calldata shown in legacy wallet popups. This alone is a huge UX improvement — users can finally understand what they're signing.
  • For enterprise customers (custodians, treasuries), the Blockaid API can be queried directly as part of a co-signing policy: "only auto-approve transactions Blockaid rates BENIGN." This is the cosigning use case.
Step 5 of 10
6
🦊 Wallet shows confirmation popup

Wallet renders the confirmation popup to the user

Now — and only now — does the user see the transaction confirmation screen. It includes Blockaid's risk label, the decoded description of what will happen (e.g. "Send 100 USDC to Uniswap"), and the expected asset changes. The user has not yet signed anything.
What happens in detail

The wallet takes the Blockaid API response from Step 5 and renders it as a confirmation popup. For BENIGN transactions, the user sees a clean screen with a green "This looks safe" badge, a plain-English description of the transaction ("Swap 100 USDC → ~0.042 ETH on Uniswap"), and the exact asset changes ("You will send: 100 USDC. You will receive: ~0.042 ETH").

For WARNING transactions, the popup shows a yellow caution header with the specific reason for concern. The user can still choose to proceed, but is clearly informed. For MALICIOUS transactions, most wallets block the popup entirely and show a full-screen red warning — the user must explicitly click "I understand the risk" to override.

Without Blockaid, the wallet popup would show raw hex calldata, a contract address, and a gas estimate — incomprehensible to 99% of users.

This is the last moment the user has full control

Once the user clicks "Confirm" in Step 7, the wallet signs the transaction and it proceeds to broadcast. The user cannot cancel after this point (unless the mempool is very slow and they manage to replace the transaction with a higher gas price). Step 6 is the informed decision point — Blockaid's job is to make this decision as clear as possible.

Step 6 of 10
7
👤 User → 🦊 Wallet

User clicks "Confirm" — wallet signs the transaction

The user approves the transaction. The wallet retrieves the user's private key (from secure storage, hardware wallet, or MPC system) and cryptographically signs the transaction. The result is a signed transaction — a raw hex blob that proves the owner authorized this specific action.
What happens in detail

Clicking "Confirm" triggers the wallet to sign the transaction using the user's private key. This produces an ECDSA (Elliptic Curve Digital Signature Algorithm) signature — three values: v, r, and s. These are appended to the transaction object to create a "raw signed transaction" — a complete hex-encoded payload ready to broadcast.

The private key never leaves the wallet. For software wallets (MetaMask), it's encrypted in browser storage. For hardware wallets (Ledger), it lives on a secure chip and the signing happens on the device. For MPC wallets (e.g. Fireblocks), the key is split across multiple servers and combined using cryptographic protocols without ever being fully assembled in one place.

Once signed, the transaction hash (txHash) is deterministic — it's the keccak256 hash of the signed transaction bytes. This is the tx hash you see on Basescan/Etherscan.

What a signed transaction contains
// Signed transaction fields (RLP-encoded for broadcast) { nonce: 0x2a, // tx count for this address — prevents replay gasPrice: 0x3B9ACA00, // or maxFeePerGas + maxPriorityFeePerGas (EIP-1559) gasLimit: 0xEA60, // max gas allowed to: 0xa0b869..., // USDC contract value: 0x0, data: 0xa9059cbb..., // encoded function call chainId: 8453, // Base network — prevents replay on other chains v: 0x1, r: 0x7f3d..., s: 0x2a9b... // ECDSA signature } // txHash = keccak256(RLP-encoded above) = 0x7db3...
Nonce
Replay protection
Sequential counter for each address. If nonce=42, this is your 43rd transaction. A transaction with a used nonce is rejected. Prevents replaying old signed transactions.
ChainId
Cross-chain protection
Ethereum=1, Base=8453, Polygon=137. Including chainId in the signature means a tx signed for Base can't be replayed on Ethereum. EIP-155 protection.
💼 SE Talking Point
  • For Safe multisig (Gnosis Safe), "signing" at this step doesn't immediately broadcast — it adds one off-chain EIP-712 signature to a pool. Only when the threshold is reached (e.g. 2 out of 4 signers) does the final execTransaction() call get broadcast on-chain.
  • For EIP-4337 Smart Accounts, the "signature" is a UserOperation, not an EOA-signed tx. It goes to a bundler, not a regular RPC node. Blockaid integrations cover both paths.
Step 7 of 10
8
🦊 Wallet → ⚡ RPC Node

Wallet broadcasts the signed tx via eth_sendRawTransaction

The wallet sends the fully signed transaction to an RPC node using eth_sendRawTransaction. This is different from Step 1's eth_sendTransaction — the transaction is now signed and ready to execute. The RPC node validates the signature and forwards the transaction to the network's mempool.
What happens in detail

The wallet calls eth_sendRawTransaction with the complete RLP-encoded signed transaction as a hex string. The RPC node performs basic validation: is the signature valid? Is the nonce correct? Is the sender's ETH balance sufficient to cover gas? If all checks pass, the node accepts the transaction and propagates it to the peer-to-peer network.

The RPC node immediately returns the txHash — the unique identifier for this transaction. At this point the transaction exists in the network but is not yet confirmed. It is "pending".

The wallet updates its UI to show a "Transaction pending" status with the txHash, often with a link to the block explorer (Etherscan, Basescan) where the user can watch it progress from "Pending" to "Success".

eth_sendRawTransaction call
// Wallet sends signed tx to RPC node { "method": "eth_sendRawTransaction", "params": [ "0x02f87282210a...full RLP-encoded signed tx hex..." ] } // RPC node responds immediately with tx hash { "result": "0x7db39872...the transaction hash..." }
eth_sendRawTransaction vs eth_sendTransaction
Key difference
eth_sendTransaction (Step 1) sends an unsigned tx to the wallet and asks it to sign. eth_sendRawTransaction (Step 8) sends a pre-signed tx to the node for broadcast. The node can't sign — it just validates and propagates.
Point of no return
Transaction is public
Once broadcast, the transaction is in the public mempool and visible to all nodes and MEV bots. It cannot be recalled. It can only be "replaced" by a new transaction with the same nonce and higher gas.
🔍
MEV (Maximal Extractable Value) exposure begins here

The moment your transaction hits the mempool, sophisticated bots (MEV searchers) can see your swap intent. They may insert a transaction before yours (front-running) to buy the token you're buying, pushing up the price — then sell it after your tx executes (sandwich attack). This is why Uniswap shows slippage warnings and offers "private" routing through MEV-protected RPC endpoints like Flashbots Protect.

Step 8 of 10
9
⚡ RPC Node → ⛓ Blockchain

Transaction enters the mempool → validator picks it up

The signed transaction propagates through the peer-to-peer network to all nodes' mempools — the public waiting room of pending transactions. A validator (Ethereum) or sequencer (Base/L2) selects it, orders it within a block, and executes it against the current world state.
What happens in detail

The mempool (memory pool) is a public, unordered set of valid-but-unconfirmed transactions sitting across all nodes in the network. When your transaction enters the mempool, it is immediately visible to anyone watching — block explorers, MEV bots, other nodes.

On Ethereum L1, validators (post-Merge, formerly miners) select transactions from the mempool to include in the next block, typically prioritising by gas price (those paying more tip get included first). Block time is ~12 seconds.

On Base (L2), a centralised sequencer (operated by Coinbase) orders all transactions, batches them, and posts them to Ethereum L1 for finality. The sequencer provides fast "soft confirmation" (milliseconds) but L1 finality takes minutes. This is why Base transactions feel instant but technically aren't "final" until the batch is posted to Ethereum.

⚡
Why sequencers matter to Blockaid customers

L2 sequencers are a potential point of centralisation and censorship — the sequencer operator can theoretically delay, reorder, or exclude transactions. For enterprise customers (exchanges, custodians, protocols), Blockaid's monitoring layer watches for anomalous transaction ordering patterns and can alert when a user's transaction is repeatedly excluded from blocks — a sign of targeted censorship.

💼 SE Talking Point
  • The mempool is where Blockaid's post-broadcast monitoring products have value. Once a transaction is signed and in the mempool, pre-confirmation protection (Steps 2–5) has already done its job. Post-confirmation monitoring watches for suspicious patterns on-chain.
  • MEV sandwich attacks happen entirely in the mempool. A bot sees your Uniswap swap, submits a buy before it and a sell after — all within the same block. Slippage tolerance is your only defence at this stage.
Step 9 of 10
✓
⛓ Blockchain confirms the transaction

TX CONFIRMED — state updated, events emitted, logs recorded

The transaction is included in a block and executed by all nodes. The world state is permanently updated. Events are emitted by the smart contract and stored as logs. The transaction hash is now findable on any block explorer. This action is irreversible.
What happens in detail

When a validator includes your transaction in a block, every full node in the network re-executes it to verify the result. The EVM runs the contract bytecode, applies the state changes (updating all relevant storage slots), and records the transaction receipt. The world state is permanently updated — balances change, allowances change, NFT ownership changes.

Any emit statements in the Solidity code fire during execution and produce event logs. These logs are permanently stored on the blockchain and are how dApps, block explorers, and monitoring tools (like Blockaid) can understand what happened inside a transaction. They're indexed by Topics for fast filtering.

The block explorer now shows the transaction as "Success" with its full receipt: gas used, block number, timestamp, all event logs, and internal call trace. This is the final, immutable record.

Transaction receipt structure
// Transaction receipt — permanently stored on-chain { "transactionHash": "0x7db39872...", "blockNumber": 18523401, "status": "0x1", // 0x1 = success, 0x0 = reverted "gasUsed": 184721, "logs": [ { "address": "0xa0b86991...", // USDC contract "topics": [ "0xddf252...", // Topic[0] = Transfer(address,address,uint256) event signature "0x000...YourAddress", // Topic[1] = indexed 'from' "0x000...UniswapPool" // Topic[2] = indexed 'to' ], "data": "0x000...0x5F5E100" // non-indexed amount (100 USDC) } // ... more logs for each event emitted ] }
Events and Topics explained

Topics are the indexed, searchable part of an event log. Topic[0] is always the event signature hash (e.g. keccak256("Transfer(address,address,uint256)") = 0xddf252...). Topics 1–3 are indexed parameters — Solidity lets you mark up to 3 parameters as indexed, making them searchable via eth_getLogs filters.

Data contains the non-indexed parameters — ABI-encoded but not searchable. For a Transfer event: sender and recipient are indexed (Topics), but the amount is in Data (still readable, just not filterable by value).

Block explorers decode these logs using the contract ABI and present them as "Event Logs" — the tab you check on Basescan to verify a swap executed correctly.

Irreversibility
Cannot be undone
Once in a confirmed block, the transaction is permanent. The only way to "undo" a transfer is if the recipient sends it back voluntarily. This is why pre-confirmation security (Steps 2–5) is Blockaid's core value.
Block finality
When is it truly final?
Ethereum: ~2 epochs (~6.5 min, 32 blocks). Base: when the batch is posted and verified on Ethereum L1 (~7 min). 1 confirmation is often enough for low-value txs; exchanges may wait for 12–64.
