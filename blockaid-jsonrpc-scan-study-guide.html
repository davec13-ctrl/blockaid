1. The big picture

The POST /v0/evm/json-rpc/scan endpoint is Blockaid's pre-confirmation transaction safety check for EVM wallets. Its single job: take a raw JSON-RPC payload that a dApp asked the wallet to sign or send, and return two things — a verdict (validation) and a simulated outcome (simulation) — so the wallet can warn the user before they commit.

Two ideas drive everything else:

  • The wallet is a dumb forwarder. It doesn't decode payloads, doesn't decide which RPC methods are "risky," doesn't try to identify attacks. It scans every signing prompt and renders the result.
  • Blockaid is the brain. It recognizes payload shapes (Permit, Permit2, Seaport order, plain transaction, UserOp), runs validation heuristics, and forks mainnet to simulate outcomes.
The endpoint is designed to be called once per signing prompt. The wallet's UX flow: dApp → wallet receives RPC call → wallet hits Blockaid → wallet renders confirmation modal with verdict → user confirms or cancels.

2. Wallet integration model

Wallets intercept every JSON-RPC method that produces a user-visible signing prompt. There is no menu of "risky" methods to pick from — the answer is always "scan all of them." The full set:

MethodWhat it isWhy scan
eth_sendTransactionStandard EOA transactionMost common attack surface (approvals, swaps, mints)
eth_sendRawTransactionPre-signed transaction broadcastSame risk profile as above
eth_signTypedData_v4EIP-712 structured signaturePermits, Permit2, Seaport orders — the off-chain attack surface
personal_signPlain message signatureSIWE phishing, log-in tricks
eth_sendUserOperationEIP-4337 account abstractionSmart-contract wallets (Safe, Argent, Coinbase Smart Wallet)
wallet_sendCallsEIP-5792 batched callsModern atomic batches
The "many examples" in the API docs dropdown aren't implementation choices for the wallet. They're a catalog of payload shapes Blockaid handles. The wallet just forwards whatever the dApp produced.

3. Request envelope anatomy

The outer shape is identical for every scan — only the inner data changes:

{
  "chain":           "ethereum",                    // Network identifier
  "account_address": "0x...",                       // The user (signer)
  "metadata": {
    "domain": "https://dapp.example"                // dApp URL → phishing check
  },
  "block":           "latest",                      // Optional pin
  "options":         ["simulation", "validation"],  // What you want back
  "data": {                                         // Raw RPC payload from dApp
    "method": "eth_sendTransaction" | "eth_signTypedData_v4" | ...,
    "params": [ ... ]                               // Whatever the method requires
  },
  "state_override":  { ... }                        // Optional: hypothetical state
}

The data field is passed through untouched from the dApp. The method inside it tells Blockaid which parser to use. The options array is the integrator's lever for cost vs. depth.

4. Why pick options selectively

A naive integration always asks for both validation and simulation. A thoughtful one picks per context:

OptionCostWhat it gives youSkip when…
validation Cheap, fast (DB lookups + rules) Verdict (Benign/Warning/Malicious), reason, features Almost never
simulation Expensive (mainnet fork + EVM exec) Asset diffs, exposures, account summary, USD impact personal_sign of plain text; quick pre-checks
gas_estimation Moderate Estimated gas + fee for the tx Wallet does its own estimation
events Moderate Decoded event log Not surfacing event detail in UI

Common pattern: validation only for low-latency pre-checks (wallet connect, hover-preview), both for the actual confirm modal where user is committing.

5. Three example requests

The same envelope wrapping three different attack shapes.

5.1 Malicious approve() transaction

dApp asks wallet to send a tx granting unlimited USDC to a drainer.

{
  "chain": "ethereum",
  "account_address": "0x5E1a0D484c5F0DE722E82f9dca3A9d5A421d47cB",
  "metadata": { "domain": "https://claim-arbitrum-airdrop.xyz" },
  "options": ["simulation", "validation"],
  "data": {
    "method": "eth_sendTransaction",
    "params": [{
      "from": "0x5E1a0D484c5F0DE722E82f9dca3A9d5A421d47cB",
      "to":   "0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48",
      "data": "0x095ea7b3000000000000000000000000a9cfc982a66d0134fee2539cfd633559fb5cf9a0ffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff",
      "value": "0x0"

}

Selector 0x095ea7b3 = approve(spender, amount). Spender is the drainer; amount is MAX_UINT256.

Expected verdict Malicious · exposures populated · assets_diffs empty.

6. What happens in Blockaid's backend

For a request with both simulation and validation on, e.g., an eth_signTypedData_v4 Seaport order:

  1. Ingress & envelope parsing. Auth check, rate limit, parse envelope. Two parallel pipelines kick off.
  2. Payload recognition. Inspect primaryType + verifyingContract to identify the shape (Permit / Permit2 / Seaport / SafeTx / custom).
  3. Validation pipeline (parallel). Domain reputation, address reputation on every involved address, schema heuristics (e.g., MAX_UINT permits, sub-floor consideration), signer context. Aggregate into verdict.
  4. Simulation pipeline (parallel). Translate the signature into the hypothetical on-chain call it would enable (Permit → permit + drain; Seaport order → fulfillOrder). Fork mainnet at current block. Execute. Decode events. Price assets in USD. Aggregate per address.
  5. Merge & respond. Combine both pipelines into the response JSON. Cross-check (e.g., catastrophic loss + Benign verdict = anomaly).

The split matters because validation is cheap and fast (DB lookups, rules) while simulation is expensive (forked EVM execution). Running them in parallel keeps total latency dominated by simulation alone.

7. The simulation response — field by field

Skip the following per the exercise: contract_management, session_key, params.user_operation_calldata.

status

"Success" means the rest of the simulation block is reliable. "Failure" means simulation reverted — degrade your UI to a "couldn't simulate, proceed cautiously" state. Read this field first.

assets_diffs

Map keyed by address. Each entry has in and out arrays of asset-movement objects. Each movement contains:

  • asset — type (ERC20/721/1155/NATIVE), address, symbol, decimals, logo
  • value — human-readable amount string
  • raw_value — wei-level string (use for math)
  • token_id — NFT only
  • usd_price — live USD per unit
  • summary — pre-rendered sentence ("Sent BAYC #5312")

Tracks what moves now. Silent on future risk.

transaction_actions

Top-level array of action labels: "approval", "transfer", "native_transfer", "token_transfer", etc. Used for icons and section routing in your UI. Note: "approval" in this list does not necessarily mean a NEW approval is granted — it can also mean an existing approval is being used. Cross-reference with exposures to disambiguate.

total_usd_diff

Per-address bottom-line dollars. { in, out, total }. The total on the user's address is the single most powerful number to surface in a confirmation modal.

exposures

Future risk granted, not present movement. Map keyed by address. Each entry has an exposures array with:

  • asset — what can be moved
  • spender — who's getting permission
  • exposure — amount, is_approved_for_all flag, token_id
  • summary — pre-rendered ("Approved unlimited USDC to 0xa9cf…")
Empty exposures ≠ safe. A tx using a pre-existing approval (Seaport farming) shows nothing here but still drains you.

total_usd_exposure

Dollar-value sibling of exposures. Per-address USD totals of new permissions granted. Lets you say "$50,000 at risk" even when nothing moves yet.

address_details

Map of every involved address → metadata: name_tag (e.g., "Seaport 1.5", "Fake_Phishing7802"), address_type, contract_name, label. Always render label substitutions for raw hex addresses in your UI.

account_summary

Same info as assets_diffs + exposures, re-centered on the user. Two sub-fields:

  • balance_changes — per-asset before/after for the user. Cleanest data to feed your "what's happening to you?" panel.
  • traces — low-level call traces. Debug / power-user only.

params

Echo of what was simulated: block, account_address, and most importantly calldata — the decoded function call (e.g., "fulfillOrder(...)"). Useful for power-user details and audit logs.

missing_balances

Diagnostic. Non-empty means simulation couldn't fully resolve some balances — treat as a confidence caveat.

simulation_run_count

Internal metric. Ignore unless debugging.

8. Why exposures matters

exposures exists because approval-based attacks move nothing in the current transaction — they're invisible in assets_diffs. Without this field, the wallet would show "nothing happens here" on exactly the transactions most likely to drain a user.

What it lets the wallet do

  1. Surface risk on tx where nothing moves. Ice-phishing approve(MAX_UINT) shows nothing in assets_diffs — only exposures reveals the danger.
  2. Separate "moves now" from "could move later" in the modal. Two different mental models for the user. Two structured fields, two UI sections.
  3. Quantify future risk in dollars via total_usd_exposure. Users understand dollars; they don't understand MAX_UINT256.
  4. Apply different UX friction to approvals vs transfers. Many wallets add typing-based confirmation or cooldown timers when exposures is non-empty.
  5. Distinguish new approvals from uses of existing approvals. transaction_actions can't tell these apart; exposures can. Only newly-granted approvals appear here.
  6. Feed post-tx "active approvals" dashboards. Wallets like Rabby / Revoke.cash use historical exposures as the canonical log of what users have authorized.
  7. Match attack class to UX treatment. The combination of assets_diffs and exposures populated/empty is a fingerprint for the attack shape (see §10).
Mental model:
assets_diffs answers "what changes if I sign this right now?"
exposures answers "what doors am I leaving unlocked after I sign?"

9. Response reading order cheat sheet

For any new response, read in this order:

  1. simulation.status — is the rest trustworthy?
  2. validation.result_type — what's the verdict?
  3. simulation.account_summary.balance_changes — what happens to the user?
  4. simulation.total_usd_diff[user_address].total — net dollar impact.
  5. simulation.exposures[user_address] + total_usd_exposure — future risk granted.
  6. simulation.address_details — who are the counterparties really?
  7. simulation.transaction_actions — what kinds of actions (UI icons)?
  8. simulation.assets_diffs — full per-address breakdown if needed.
  9. simulation.params.calldata — decoded function call (if you care).

11. Case study: Seaport farming attack flow

The setup (months earlier, dormant)

Victim once listed an NFT on OpenSea, which required calling setApprovalForAll(Seaport, true) on the BAYC contract. This grant is permanent and gives Seaport authority to move any BAYC the victim owns — forever. This is the loaded gun.

The attack (the moment of the scan)

  1. Victim lands on a phishing site (Discord link, fake airdrop, fake staking page).
  2. Site requests eth_signTypedData_v4 — a signature, not a transaction. No gas, no obvious red flag in MetaMask.
  3. The typed data is a Seaport order: "I offer BAYC #5312 for 0.001 ETH, anyone can fulfill."
  4. Victim clicks Sign. Nothing visible happens.
  5. Attacker submits the signed order to Seaport, paying 0.001 ETH themselves.
  6. Seaport validates: signature good, approval exists, payment supplied. Atomically transfers BAYC out and 0.001 ETH in.
  7. Victim is left with 0.001 ETH ($1.65). Attacker has a $39,859 NFT.

What the response shows

  • validation.result_type: "Malicious", reason: "seaport_farming"
  • assets_diffs — three addresses: victim (BAYC out, ETH in), attacker helper (ETH out, decoy payment), attacker receiver (BAYC in, tagged Fake_Phishing7802)
  • total_usd_diff — victim total ≈ −$39,857.65
  • exposures: {} — empty, because no NEW approval is granted; the old setApprovalForAll is being used
  • transaction_actions: ["approval", "native_transfer", "token_transfer"] — note "approval" appears even though exposures is empty (use vs. grant)
  • address_details — Seaport 1.5 labeled, attacker tagged as known phishing
The malicious step is a signature. The damage is realized later, on-chain. Without simulation, the user has no way to see the economic reality.

12. Self-check questions

Q1. A user signs an EIP-2612 Permit granting unlimited USDC to a drainer. Which fields populate, which stay empty?
A: exposures and total_usd_exposure populate (large $). assets_diffs and total_usd_diff are empty. Mirror image of Seaport farming.
Q2. simulation.status: "Failure" but validation.result_type: "Malicious". What should the wallet display?
A: Show the Malicious verdict prominently — validation is independent of simulation. Do not show assets_diffs, exposures, or total_usd_diff values; they're unreliable on Failure. Render: "Blockaid flagged this as malicious. We couldn't simulate the outcome."
Q3. transaction_actions contains "approval" but exposures is empty. What does that mean?
A: The transaction uses an existing approval rather than granting a new one. Classic Seaport farming pattern.
Q4. Why might a wallet ask for validation only and skip simulation?
A: Latency-sensitive pre-checks (wallet connect, tooltip preview), low-cost rate-limit management, or a method like personal_sign on plain text where simulation has nothing meaningful to compute.
Q5. The single most powerful number to display in a confirmation modal?
A: total_usd_diff[user_address].total — the user's net dollar impact in one number.
Q6. The wallet receives eth_sendUserOperation from a Safe. Does it scan?
A: Yes. Every method that produces a user-visible signing prompt gets scanned. UserOps included.
