Solana 101

Solana Tokens, Explained Simply

A plain-English guide to how tokens work on Solana, who has control, and what happens when things go wrong.

The Architecture at a Glance

A token's "anatomy" on Solana: a program, two account types, and five roles.

Think of this as a bank for digital tokens. The Token Program is the bank itself — it sets the rules. The two account types are different kinds of records the bank keeps. And the people are roles with specific permissions.

The Building Blocks

Every Solana token is built from these pieces. Here's what each one does in everyday language.

Token Program

The institution that runs the show. Sets the rules for how all tokens work — who can mint, transfer, freeze, etc. One Token Program serves every token on Solana.

Token Mint Account

The "headquarters" for one specific token. Defines what the token is, how many exist, and its basic properties. One per token type (USDC, BONK, etc.).

Token Account

Your personal wallet for one specific token. To hold USDC and BONK, you need two separate token accounts. Each one points back to its mint account.

Mint Authority

The only entity that can create new tokens. Like a central bank with a printing press.

Freeze Authority

Can freeze any wallet holding the token, preventing transfers. Useful for stablecoins; dangerous in the wrong hands.

Token Owner

You. Holds the tokens in a token account and can spend them.

Close Authority

Can shut down a token account when it's empty and no longer needed (and reclaim the small SOL deposit).

Delegate

Like limited power of attorney — can spend up to a set amount on the owner's behalf. Common when interacting with apps.

What Happens If a Bad Actor Gets Control?

Each role carries a different blast radius. Some compromises affect one user; others can wipe out an entire token.

!

Mint Authority Compromised — Catastrophic

Attacker prints unlimited tokens, dumps them on exchanges, and crashes the price to near zero. Token holders are left with worthless bags. Known as an "infinite mint attack."

!

Freeze Authority Compromised — Severe

Attacker can freeze any wallet, locking funds in place. Could be used for extortion, targeted attacks, or to make the token entirely unusable.

!

Token Owner Key Stolen — Bad for One User

The classic wallet hack. Attacker drains that wallet, but other holders are unaffected. Hardware wallets and good security hygiene are the defenses here.

!

Delegate Compromised — Limited

Damage capped at whatever was delegated. But "unlimited approvals" to apps are common and dangerous — a compromised app can drain everything you approved.

i

Close Authority Compromised — Limited

Can force-close empty accounts. Annoying and can grief users, but doesn't directly steal funds.

How Solana Defends Against Mint Attacks

Solana itself doesn't block minting — that's a legitimate feature. Protection comes from a stack of practices and tools projects use on top.

1

Renounce the Mint Authority

Set the authority to "None" — permanently, irreversibly. The printing press is destroyed. Most fixed-supply tokens do this. Infinite mint attack becomes mathematically impossible.

2

Multi-Signature Wallets

For tokens that must keep minting active (stablecoins), the authority is held by a multi-sig requiring 3-of-5 or similar. An attacker has to compromise multiple independent people at once.

3

Timelocks

Even after a multi-sig approves a mint, the action sits in a queue for hours or days. Gives the community time to spot suspicious activity and react before it executes.

4

Program-Controlled Authorities

The mint authority is a smart contract with hard-coded rules — e.g., "only mint when a real dollar deposit is verified." Trust shifts from people to auditable code.

5

The Security Ecosystem

Wallets (Phantom, Backpack), explorers (Solscan), monitoring services, and security providers like Blockaid scan tokens, simulate transactions, and warn users about risky setups before they sign.

The Bottom Line

Solana tokens are powerful because they're flexible — but flexibility means trust assumptions. A token with a renounced mint authority is the safest from supply attacks but can never adapt. A token with an active mint authority needs strong custody (multi-sig + timelock + audits + monitoring) to be safe. Before interacting with any token, the most important question is simple: who controls the authorities, and what could they do if they wanted to — or if they got hacked?
